What this is

claude-monitor is a local daemon that watches your Claude Code activity and surfaces it in this dashboard. It does two things in parallel:

  1. Statusline snapshots — Claude Code calls cm statusline as a statusLine hook. Every call ships the raw JSON (cost, context window, rate limits) to the daemon.
  2. Transcript tail — a file watcher on ~/.claude/projects/**/*.jsonl reads new assistant lines as they're written and records per-turn token counts. This works even without the statusline hook.

All data lives in ~/.claude/claude-monitor/monitor.db (SQLite, WAL). The UI streams updates from /v1/live over a WebSocket.

Getting data flowing

  • On macOS: cm install registers a LaunchAgent for the daemon and wires up the statusLine hook in ~/.claude/settings.json. Run cm uninstall to reverse it.
  • On other platforms: start cm-app (or cm-app-tauri) yourself, and add cm statusline as a statusLine hook in your Claude Code settings.
  • The daemon binds 127.0.0.1:0 and writes the chosen port to ~/.claude/claude-monitor/port. Pin one with CM_PORT=NNNN.
  • If the daemon isn't running when the statusline fires, payloads are appended to ~/.claude/claude-monitor/queue.jsonl and drained on next start — no data lost.

Views

Live
Cards for the active session (model, project, context-window %, session cost, 5h/7d limits) plus a per-turn token chart. Updates in real time as turns and snapshots stream in.
Sessions
Every session the daemon has seen. Filter by time range and model, sort any column, hide columns via the Columns ▾ button, resize columns by dragging the right edge. Click a row to drill into that session.
Trends
Daily cost (USD) and daily total tokens, bucketed server-side from /v1/trends?window=day.
Session detail
Opened by clicking a row in Sessions. Shows totals, per-turn stacked tokens, snapshot timeline (cost / context% / 5h%), and any subagents that ran.

Column reference

Hover any column header or card for a tooltip. The ones worth calling out:

Quota tokens
Input-normalized count used for 5h/weekly quota accounting: input×1 + output×5 + cache_read×0.1 + cache_write_5m×1.25 + cache_write_1h×2. This weighting mirrors how Anthropic publicly describes rate-limit math. The % 5h / % weekly suffix is derived by inverting used_percentage from the latest statusline snapshot against the matching turn-token window.
Cost (snapshot) vs Cost (est.)
Snapshot is whatever Claude Code's statusline reported last (cost.total_cost_usd). Estimated is computed locally by pricing each turn against cm-core's pricing table. Snapshot is empty if the statusline hook never ran; estimated is empty for unknown model families (no generic fallback, by design).
Cache R / Cache W 5m / Cache W 1h
Prompt-caching token counts. Reads are the cheapest (0.1×), 5-minute writes sit in the middle (1.25×), 1-hour writes are the most expensive (2×). Unattributed cache-creation bytes are counted toward the 5m bucket.
Last seen
Most recent turn or snapshot timestamp. Drives the default sort and the time-range filter.

HTTP surface

The daemon exposes a small JSON API on the same port as this UI:

  • GET /v1/health — liveness
  • GET /v1/sessions — rollups
  • GET /v1/sessions/:id/turns — per-turn rows
  • GET /v1/sessions/:id/snapshots — statusline snapshots
  • GET /v1/sessions/:id/subagents — subagent tasks
  • GET /v1/trends?window=hour|day|week — bucketed totals
  • GET /v1/quota-caps — estimated absolute 5h/weekly caps
  • GET /v1/live — WebSocket, broadcasts snapshot / turn / subagent_snapshot events
